# Demo checklist

## Ninety-second structure

**0–15 seconds — problem:** identify the user and painful decision with one piece
of evidence.

**15–30 seconds — before:** show the fragmented/noisy/late input.

**30–70 seconds — product:** complete the golden path without detours.

**70–85 seconds — value:** show the measurable operational or resident/student
benefit and explain why the recommendation/rank is understandable.

**85–90 seconds — boundary:** state what the prototype does not automate and the
next pilot step.

## Production check

- [ ] Live URL opens in a private window.
- [ ] `/api/health` returns `status: ok`.
- [ ] Primary path works at phone width.
- [ ] No real sensitive data appears.
- [ ] No browser console error on the golden path.
- [ ] Keyboard focus is visible and order is logical.
- [ ] All demo inputs are preloaded or short.
- [ ] Offline screenshots/recording are open locally.
- [ ] A known-good rollout can be restored.

## Presenter rule

Do not narrate the stack before showing the problem and outcome. Technology is
evidence of feasibility; it is not the public value.
